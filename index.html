  • I n t r o d u c t i o n   ...top

    The GUI Websphere Admin Console is the default tool for administrative tasks.  
    Alternatively, the Websphere Admin Utility can be used to execute many of these tasks from the command line.

    • Perspective   ...top
      Depending on the perspective, admin commands offer different advantages:
      • For Support, speed & simplicity. admin commands are executed from a telnet session
      • For Development, continuous & consistent deployment across environments
      • For Quality Assurance, automated deployment is the precursor to automated testing
      • For Administration, scripting of mundane & repetitive tasks
      • For Management, fine-grain control of roles & responsibilities

  • J a v a   V i r t u a l   M a c h i n e s (JVM)   ...top

    Each application server has a Java Virtual Machine (JVM).   JVM system properties (JVMSP) are building blocks with which an application server JVM is configured.   There are 500 JVMSPs within each environment to administer.  

    • JVM System Properties (JVMSP)   ...top

      The JVM system property (JVMSP) declares a property name=value pair on the java command line with the "-D" option.   JVMs typically read their JVMSPs upon startup, for example: 
      This program:
   public class getSystemPropertiesExample {    
     static public void main(String[] args) {        
       System.getProperties() . list( System.out ) ;
     }
   }

With this command line:
   java -DPORTAL_URL="http://maxtrad.com" getSystemPropertiesExample 

Produces this output:
   PORTAL_URL="http://maxtrad.com"

      JVMSPs are configured for individual servers, for the server group, or both.   Different server groups follow different configuration policies within the same environment.   No single policy is optimal for all application servers.   These complexities pose a challenge to JVMSP change and administration.  

      The Websphere Admin Console is the default JVMSP editing tool.   It does not provides a global view of related JVMSPs.   Viewing all the related JVMSPs with the Websphere Admin Console requires multiple round trip fetches to the repository database.   Editing the JVMSP with the Websphere Admin Console is time-consuming and error prone.   Resulting JVMSP changes can be in-consistent and require additional verification.   Backing up the JVMSPs is recommended beforehand.   Re-starting the server is required after each iteration.   Verification, backup and re-start each extend the time required to update JVMSPs using the Websphere Admin Console.  

    • JVMSP XSLT Editors   ...top

      The admin command utility includes 2 JVMSP XSLT editors to address these challenges.   For a given server, the JVMSP XSLT editors

      • Export the server's configuration to an XML document
      • Backup the server's configuration in case of rollback
      • Transform the server's configuration using an XSLT processor & templates
      • Import the transformed configuration back into the server

      The JVMSP XSLT editors use the following command-line parameters:  

      1. Server name.   Changes are applied to only this server
      2. Update type.   Valid update parameter values:
        • 'set' for change.   properties are changed
        • 'del' for delete.   properties are deleted
      3. Update scope.   Valid scope parameter values:
        • a specific server
        • 'servers' for all servers in the group
        • 'group' for only the group and not the servers
        • 'ALL' for all servers in the group as well as the group
      4. Property name
        • names may not contain white space
        • names may not contain special characters such as :"@#&^+|~!$*'/.?
      5. Property value
        • if 'properly quoted', values may contain white space
        • if 'properly quoted', values may contain special characters such as :"@#&^+|~!$*'/.? 
       
usage: was4nd jvmspEdit $server  $op  $scope  $property  $value
       |      |         |        |    |       |          |
       |      |         |        |    |       |          \- property value
       |      |         |        |    |       \------------ property name
       |      |         |        |    \-------------------- update scope 
       |      |         |        \------------------------- update type
       |      |         \---------------------------------- server name
       \      \-------------------------------------------- jvmspEdit command 
       \--------------------------------------------------- websphere command utility name

    • batch changes   ...top

      Batch JVMSP changes are executed with 3 commands - jvmspBatchExport, jvmspBatchEdit and jvmspBatchImport.   The jvmspBatchEdit command requires the same command-line parameters listed above.   In the previous example, note that jvmspEdit actually calls the batch commands  

      Batch commands are most useful in two scenarios:

      • To make multiple changes.   Export once, make multiple JVMSP edits, and import once.
      • To Review changes before import. 
       
perform this just once ... usage: was4nd jvmspBatchExport $server  
perform this "N" times ... usage: was4nd jvmspBatchEdit $server  $op  $scope  $property  $value
perform this just once ... usage: was4nd jvmspBatchImport

    • Other Notes
    • In contrast to the other admin commands, the JVMSP editors are not yet functional in multi-user mode.  
    • One of the generated xml files includes the all server properties.   This can be used as a template with which any server property can be edited and imported.  

    • List of generated files.   Example: TPT
    • TPT-export1.xml.040212173659 - time-stamped backup xml configuration
    • TPT-export1.xml - complete XMLConfig export for one application sever
    • TPT-export2-clean.xml - cleaned up, smaller than original, easier to edit, view and print
    • TPT-edit0.xml - xml document after first transformation
    • TPT-editN.xml - xml document after N transformations
    • TPT-import1.xml - xml document to be imported

  • A p p l i c a t i o n s   ...top

    Applications are installed from J2EE Enterprise Application aRchive (EAR) files. Applications are deployed to Server groups. Server groups are composed of application servers.

  • Network Deployments   ...top

    Network deployments require an agent to be running on the remote node(s). Applications are deployed from teak and the agent runs on elm.

    The network deployment functions have an email option. The deployment log is emailed to the address supplied as the last parameter.

    • EAR Install   ...top
      The ear install
      • backs up the current application
      • stops the server group
      • deletes the current application
      • installs the new application
      • assigns a virtual host to the web app component
      • distributes binaries to the remote node(s)
      • creates a deployment log, with an email option

      The current version of the EAR install program makes two assumptions:

      • server and virtual host names are identical. This is true of all current applications.
      • one WAR component per application. This is true of all current applications. 
       
Command line usage:
  was4nd  earInstall  Application  Server  EarFilePath [emailAddress]
        Application - e.g: MaxtradLite, CFO-Central
        Server      - e.g: TPT, CLD, DT
        EarFilePath - e.g: TPT_ver_1.2.3.ear
        [email]     - e.g: john.doe

    • EAR Install Warm   ...top
      The difference between this command, earInstallWarm, and the previous one is that
      • the enterprise application is stopped, not the application server
      • the enterprise application is re-started after the installation.

      Consequently, during a warm application installation, the servers are not stopped. This is particularly useful for those applications that contain multiple applications. It is intended for "well behaved" applications only.

        EarFilePath - e.g: TPT_ver_1.2.3.ear
        [email]     - e.g: john.doe

      The earJSPCompile command invokes the IBM JSPBatchCompiler utility. It compiles JSP files from the installedApps directory, into the temp cache directory. The IBM JSPBatchCompiler utility does not compile all JSPs, in particular those under the WEB-INF directory. There are also other issues that are not resolved without the current fixpack. Care must be taken not to over-state this tool's capabilities. Nevertheless, pre-compiling JSPs can minimize the impact of many deployments. The JSPCompile command runs as the websphere admin account. [pressc:] TODO: After this command is executed, file access rights need to be updated for the application account (e.g: dtuser).
